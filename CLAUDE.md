# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Structure

Monorepo with three independent sub-projects:

- `portfolio-site-v6-prodigy-ui/` — Vue 3 frontend (Vite, TypeScript, Pinia, Vue Router, Tailwind CSS 4)
- `portfolio-site-v6-prodigy-api/` — Spring Boot 4 backend (Java 21, JPA, PostgreSQL)
- `portfolio-site-v6-prodigy-admin/` — Astro admin panel (WebAuthn/TOTP/JWT-gated CMS covering every content type the API exposes)

## UI Commands

All commands run from `portfolio-site-v6-prodigy-ui/` using `pnpm`.

```bash
pnpm install          # install dependencies
pnpm dev              # dev server with HMR
pnpm build            # type-check + production build
pnpm test:unit        # run Vitest unit tests
pnpm test:unit -- --reporter=verbose  # run a single test file: pnpm test:unit src/components/__tests__/HelloWorld.spec.ts
pnpm lint             # oxlint then eslint (both with --fix)
pnpm format           # prettier over src/
pnpm type-check       # vue-tsc only (no emit)
```

Node engine requirement: `^22.18.0 || >=24.12.0`.

`@` is aliased to `src/` in both Vite and TypeScript configs.

## API Commands

All commands run from `portfolio-site-v6-prodigy-api/` using the Gradle wrapper.

```bash
./gradlew bootRun     # start Spring Boot dev server
./gradlew test        # run JUnit 5 tests
./gradlew build       # compile + test + package JAR
./gradlew test --tests "com.sresthaa.SomeTest#methodName"  # run a single test
```

Requires Java 21. The app expects a running PostgreSQL instance configured externally.

## Admin Commands

All commands run from `portfolio-site-v6-prodigy-admin/` using `npm` (this sub-project uses `npm`/`package-lock.json`, not `pnpm` like the UI).

```bash
npm install           # install dependencies
npm run dev           # dev server with HMR
npm run build         # production build (outputs to dist/)
npm run preview       # preview the production build locally
```

Node engine requirement: `>=22.12.0`.

Has its own `AGENTS.md`/`CLAUDE.md` (identical, both starter-generated) noting that `astro dev` should be started with `--background` and managed via `astro dev stop`/`status`/`logs`.

## Architecture

**Frontend** — Vue 3 Composition API with `<script setup>`. Styling via Tailwind CSS 4 (`@tailwindcss/vite` plugin) with `@headlessui/vue`, `@heroicons/vue`, and `@fortawesome/vue-fontawesome` (solid + brands icon sets). State via Pinia stores (`src/stores/theme.ts` — light/dark theme, persisted to `localStorage`, toggles the `.dark` class on `<html>`; custom color tokens for both modes are defined via `@theme` in `src/assets/main.css`). Routing via Vue Router 5 with web history (`src/router/index.ts`). Views live in `src/views/`, reusable components in `src/components/`.

Routes: `/` (Home), `/projects`, `/blog`, `/blog/:slug`, `/highlights`, `/contact` — each a thin View that wraps a same-named Component (e.g. `ContactView.vue` renders `ContactComponent.vue`), following the existing `HomeView`/`HomeComponent` pattern. `NavbarComponent`'s `navigation` array drives both the nav links (`RouterLink`, desktop + mobile) and `route.meta.title` per route; `isCurrent()` matches by prefix (`route.path.startsWith(item.href + '/')`), not just exact equality, so a nested route like a blog post still highlights its parent nav item. All top-level pages have real UI now.

`HomeComponent.vue` is ported from the old Next.js site: a hero `<section id="home">` (profile picture, greeting, name, bio lines, social icon row via `@fortawesome/free-brands-svg-icons`, animated scroll-down chevron that smooth-scrolls to `#about`) followed by an `<section id="about">` (intro blurb, story paragraphs, and `TimelineWheel.vue` for the career history). All the old site's hardcoded `text-white`/inline `--color-text-*` styles were replaced with the semantic tokens (`text-ink`/`text-ink-muted`, accent color → `text-primary`). The hero uses `min-h-[calc(100vh-4rem)]` rather than `min-h-screen` — the old site's Navbar was a floating overlay that didn't consume layout space, but ours is `sticky` and takes up a real `h-16` (4rem), so a plain `min-h-screen` hero would overshoot the first viewport by that much and push the scroll-down button below the fold.

**Dynamic content pattern** — every content page loads via `src/composables/useCachedResource.ts`, a generic `useCachedResource<T>(key, url, ttlMs)` that checks a timestamped `localStorage` entry first (default TTL: 1 week) and only `fetch()`s if stale/missing, falling back to stale cache on fetch failure rather than showing nothing. All content pages now point at the real Spring Boot API via `getApiBaseUrl()` (`src/utils/apiBaseUrl.ts`, backed by `VITE_API_BASE_URL`): `HomeComponent.vue` → `/api/home`, `ProjectsComponent.vue` → `/api/projects`, `HighlightsComponent.vue` → `/api/highlights`, `BlogComponent.vue`/`BlogPostComponent.vue` → `/api/blog`/`/api/blog/:slug`. Each started as a `public/data/*.json` mock (gitignored, explicitly a stand-in, never tracked content) and got swapped to the real endpoint one page at a time; the cache key was bumped (e.g. an `-v2` suffix) whenever a swap also changed the response shape (object-wrapped `{ items: [...] }` → bare array), since reusing the old key would silently read a stale, wrongly-shaped cache back as if it were the new format. `public/data/*.json` files may still linger locally from before a given swap but nothing reads them anymore — safe to ignore or delete.

**`TimelineWheel.vue`** (career history, work in progress) — a rotating radial dial rather than a static list: events sit at even angles around a full circle (`360 / count` apart), only spread across `chronological.value` (oldest-first, so it reads left-to-right as it rotates). Only labels within `VISIBLE_RANGE` degrees of the top are shown (rest are conceptually "around the back", faded via `opacity`/`pointer-events`); `rotation` increases continuously via a `requestAnimationFrame` loop, so different years drift through a **stationary** glass window (`border-primary/60 bg-primary/25 backdrop-blur-md`) fixed at the arc's topmost point. Interactions: hovering a visible label eases (spring-lerp, not a hard jump) the rotation so that label's angle converges on 0 (into the window), pausing auto-rotate until the mouse leaves; dragging (`pointerdown`/`pointermove`/`pointerup` on `window`, not just the element, so drags continue even if the pointer leaves the component bounds) directly maps horizontal movement to rotation, overriding both auto-rotate and hover-ease while held. The nearest-to-0 label is also magnified (`scale` ramping 1x→`MAX_SCALE` within `MAGNIFY_RANGE`), like it's being read through a magnifying glass. Shown from `md` up (`hidden md:block`, 768px+) — chosen over the default `lg` (1024px) because that cut off iPad-class tablets in portrait (e.g. iPad 10th-gen at 820px wide) even though there's enough width for the arc; below `md` the geometry genuinely doesn't translate, so it falls back to the horizontal auto-scroller described below. The active-event summary text (date/title/position/institution below the wheel) is gated behind the same `hidden md:block` breakpoint rather than always rendered, since it's a readout paired with the wheel and looks like orphaned clutter stacked above the mobile cards otherwise. All geometry constants are in **rem**, not px, including the ones fed into inline `transform`/`left`/`top` styles — rem is the standard unit for anything that should track the user's root font-size.

**Mobile fallback (`md:hidden`)** — a rotating arc doesn't translate to narrow viewports, and stacking all events as vertical cards made the About section extremely long, so instead the same cards scroll horizontally and auto-drift continuously (`MOBILE_AUTO_SCROLL_SPEED_PX_PER_SEC`), driven by the same `requestAnimationFrame` loop (`tick()`) as the desktop arc — it just also writes `scrollerRef`'s `scrollLeft` alongside `rotation`. The event list is rendered twice back-to-back (`loopedEvents`) so it can loop infinitely: once `scrollLeft` passes one full set's width, it's snapped back by that width, invisible since the next set is an identical duplicate. Auto-scroll pauses on `pointerdown` and resumes on the container's native `scrollend` event (see gotchas below for why not `pointerup`), with a 3s failsafe timeout in case `scrollend` never fires. Cards are `bg-surface-muted/75` (translucent, not opaque) and the scrollbar is hidden (`[scrollbar-width:none] [&::-webkit-scrollbar]:hidden`) since the auto-drift and swipe gesture already make scrollability obvious without one.

Gotchas hit building this one (worth knowing before touching it again):
- **Don't share `transform-origin` between rotate and scale.** The label's rotate (tangent-to-arc lean) uses `origin-top`; when the magnify `scale()` was on the *same* element, scaling around a top-pinned origin made the box grow downward from that fixed edge, so the text's visual center bobbed up/down continuously as it entered/left the magnify zone. Fixed by splitting into a wrapper (`<button>`, handles position + rotate, `origin-top`) and an inner `<span>` (handles `scale()` only, default centered origin).
- **`backdrop-blur` blurs whatever's *behind* it, including your own text if it's behind too.** The glass window was originally `z-10` (on top of the labels), which blurred the currently-active label's own text into an unreadable smear. Fixed by rendering the window *before* the labels in the DOM (so it sits behind them) — the blur now only softens the page background, not the number passing in front of it.
- **Don't snap an "active by proximity" element's position to a fixed target.** To fix a minor magnify-scale misalignment, the active label's rendered position was briefly forced to exactly match the window's coordinates for as long as it stayed "nearest to 0" — which visibly froze it in place for several seconds during auto-rotation instead of letting it glide through. Reverted; positions must stay driven purely by the continuous per-frame angle, never overridden by "closest to X" logic.
- A wide angle sweep (e.g. the dome-shaped predecessor of this component used `±80°`) makes the cosine curve flat on top and steep at the ends, reading as lopsided rather than a smooth arc — keep sweeps/visible-ranges narrower and gentler.
- **Don't accumulate sub-pixel scroll via `element.scrollLeft +=`.** Some browsers round `scrollLeft` to the nearest integer on read/write; at the mobile scroller's speed (~0.6px/frame at 60fps) each frame's fractional increment gets truncated straight back to the same integer forever, so it never visibly moves even though the logic looks correct. Fixed by tracking the position in a plain JS variable (`mobileScrollPosition`) and only ever assigning (not incrementing-via-readback) it to `scrollLeft`.
- **Resume mobile auto-scroll on `scrollend`, not `pointerup`.** A swipe's momentum keeps moving `scrollLeft` natively for a bit after the finger lifts; resuming immediately on `pointerup` means the very next frame's auto-scroll assignment overrides that still-decelerating native scroll, which reads as "autoscroll ignoring my swipes" rather than a smooth handoff. `scrollend` only fires once the browser's scroll (including momentum) has actually settled.

`BackgroundComponent.vue` (also ported from the old Next.js site) renders `public/assets/telescope.svg`/`laptop.svg`/`astronaut.svg` as a `fixed -z-10 pointer-events-none` background — telescope+laptop split top/bottom on the left (hidden below `md`), astronaut on the right (always visible). It's `fixed` to the viewport (not `absolute` inside `<main>`) so it stays static and doesn't stretch on tall pages (e.g. `HomeComponent.vue`'s hero+About content) — it renders behind the translucent/blurred Navbar and Footer rather than being clipped by them, since both already use `bg-surface/80 backdrop-blur-md`. It uses independent `top`/`right`/`bottom`/`left` offsets (currently more inset from the top than the other sides) rather than a symmetric `inset-*` so the illustration block can be nudged in any direction without fighting the other sides; the rotated+scaled laptop graphic (`-rotate-18 scale-[0.65]`) needs enough inset margin or its rotated bounding box clips against the container's `overflow-hidden` edge. The SVGs are solid black glyphs with no color tokens of their own, so it's rendered at low opacity (`opacity-10`) with `dark:invert` to flip them white in dark mode — adjust that pair, not the SVG files, if it needs to read more/less strongly.

**`HighlightsComponent.vue`/`HighlightCard.vue`** and **`ProjectsComponent.vue`/`ProjectCard.vue`** — ported from `portfolio-site-v5-prime` (the previous Next.js/Prisma/tRPC version, at `../portfolio-site-v5-prime`), specifically its `moments`/`gallery` router (Highlights) and `projects` router (Projects): same data shape (comma-separated `tags`/`techStacks`, `mediaType`/`statusFlag`/`sourceCodeAvailability` enums, etc.) and the same feature set (tag/tech-stack filter with staged apply, search on Projects, pagination, expandable descriptions), loaded via `useCachedResource` from the real `/api/highlights`/`/api/projects` endpoints (see the Dynamic content pattern paragraph above). Filtering/search/pagination all happen client-side over the full returned list — there's no server-side query support, deliberately kept simple for a low-traffic personal site; don't add query params to these endpoints without a real need. Status/source-code/collab-mode badges are mapped onto the existing 4 semantic tokens (`primary`/`secondary`/`warning`/`danger`) rather than the old site's raw Tailwind palette colors (`bg-green-500/30` etc.) — e.g. `COMPLETED`→secondary, `IN_PROGRESS`→warning, `ARCHIVED`→danger — since 5 enum values map cleanly onto the tokens already defined; don't introduce new raw colors for future badge-like UI without checking if an existing token fits first. `HighlightItem` has **two** image fields (`thumbnailImage` shown in the grid card, `image` shown full-size in the modal) since that pair is a deliberate product requirement, not an artifact of the port. `Project` originally only had one `image` field; it later gained an optional `mediaType: 'IMAGE' | 'VIDEO'` + `videoUrl` (mirroring `HighlightItem`'s image/video split, `mediaType` undefined treated as `'IMAGE'` for data saved before the field existed) so a project can show an embedded YouTube demo instead of a screenshot — IMAGE projects still only have the one `image` field, no separate thumbnail. The YouTube-embed-id regex is shared between `HighlightCard.vue` and `ProjectCard.vue` via `src/utils/youtube.ts` rather than duplicated.

Two pieces of logic were identical between these two pages and got extracted rather than duplicated:
- **`useUniformCardHeight.ts`** (composable) — gives a grid of cards uniform height on desktop *as a floor, not a live stretch*: card heights are measured once via a `requestAnimationFrame`-free `nextTick` pass and applied as inline `min-height`, recomputed only when the rendered card *set* changes (new page/filter/resize) — never when a single card's own content changes size (e.g. expanding a "Read more"). This matters: an earlier version used CSS Grid `align-items: stretch`, which re-stretches the whole row live off current content, so expanding one card's description dragged every card in its row taller too. Switching to `items-start` + this JS-measured floor fixes that, since the floor is decoupled from any one card's `isExpanded` toggle. Gotcha: the `watch(itemsSource, ..., { immediate: true })` is required, not optional — `useCachedResource` resolves **synchronously** from `localStorage` when its cache is fresh (no `await` before that assignment), so the watched source can already be populated *before* the watcher is even created, meaning a non-immediate watcher silently never fires on a warm cache.
- **`TagFilterPopover.vue`** (component) — the multi-select tag/tech-stack filter dropdown (headlessui `Popover`, staged `pending` selection with Apply/Clear, search-to-narrow, removable applied chips), parameterized by `label`/`panelTitle`/`searchPlaceholder`/`allValues` and a `v-model` of applied values.

Both pages use headlessui (`Popover` for the filter, `Dialog`+`TransitionRoot`/`TransitionChild` for Highlights' image modal) rather than hand-rolled dropdown/modal logic, for free outside-click/Escape/focus-trap handling — same reasoning as `NavbarComponent`'s `Disclosure`.

**Page width gotcha** — a static `max-w-*` utility (e.g. `max-w-7xl`) is a fixed upper bound: raising it only helps once the viewport is *already* wide enough to hit that bound, so bumping `6xl`→`7xl` can look like it did nothing if the real viewport (e.g. a narrowed devtools layout) was already below the old cap. Tailwind's `container` utility (`class="container mx-auto px-4 py-12"`, no `max-w-*` needed) is meaningfully different — it steps its `max-width` up through *every* breakpoint (sm/md/lg/xl/2xl, topping out at `96rem`/1536px), matching the old site's spec exactly. `ProjectsComponent.vue` uses `container`; check whether a "make it wider" request is actually hitting a real cap before reaching for a bigger `max-w-*` number.

**`BlogComponent.vue`/`BlogCard.vue`** (list, `/blog`) and **`BlogPostComponent.vue`** (detail, `/blog/:slug`) — the list follows the same search/tag-filter/pagination/`useUniformCardHeight` pattern as Projects/Highlights, ported from `portfolio-site-v5-prime`'s `blog` router (`BlogPost`/`Comment` data shape: `slug`, `excerpt`, markdown `content`, `tags`, `publishedAt`, etc.), filtered to `published: true` posts only. Interesting find while porting: the old schema already had `metaTitle`/`metaDescription`/`ogImage` fields, but grepping its routes turned up no `generateMetadata` anywhere — those fields were never actually wired up, so real link previews are a new feature this time, not a restored one. Those per-post override fields were later **removed** from `BlogPost` again, though — they always just fell back to `title`/`excerpt`/`coverImage` when absent (see `useDocumentMeta` usage in `BlogPostComponent.vue`), so in practice they only ever duplicated fields the post already had; link previews now read those three fields directly instead. Site-wide SEO fallbacks (for pages with no per-entity content of their own — Home/Projects/Highlights/Contact) live in `stores/siteSettings.ts` instead, backed by `public/data/site-settings.json` and editable via the admin's `/settings` page — see the **Site settings** paragraph below. Share button, like/dislike, and comments are **also new** — there's no precedent for them in the old site at all:

- **Share** — `useShare.ts`: native share sheet (`navigator.share`) when available, falling back to clipboard-copy with a 2s "Copied!" flag. Local-only by nature, no persistence needed regardless of backend state.
- **Like/dislike** — `useBlogReactions.ts`: real, persisted via `POST /api/blog/:slug/reactions`. There's no visitor identity on the public site by design (see below), so the *server* can't remember what a given browser voted last time — the client sends both ends of the transition (`previousVote`/`vote`) and the server blindly applies the delta (clamped at 0, never goes negative). This browser's own vote is tracked in `localStorage` (`blog-reaction-{slug}`) purely to decide which button renders highlighted, not to compute the displayed count — the count shown is always the server's authoritative response from the most recent vote (or the post's fetched `likeCount`/`dislikeCount` before this session's first vote). This split matters: layering a local `+1` on top of a base count (the old mocked approach) would double-count once a refreshed base already includes a previously-persisted vote from the same browser.
- **Comments** — `useBlogComments.ts`: real, `GET`/`POST /api/blog/:slug/comments`, newest first. Rendered via normal Vue text interpolation (auto-escaped) rather than the old site's `dangerouslySetInnerHTML`+DOMPurify pattern — safer by construction for user-authored text, no sanitization step needed client-side (the API still validates/trims length server-side).

No visitor accounts exist on the public site, and this is permanent by design (removed a few iterations ago specifically to avoid the credential-security burden of holding visitor accounts on a personal site) — consequences: no real per-visitor vote dedup (a browser can clear `localStorage` and vote again), and no admin moderation UI for comments (moderation is manual, via direct DB edits). Anti-spam on both comments and the Contact form is a known, deliberately deferred gap, not an oversight.

**Blog image library** (admin-only, `/blog/images`) — a reusable media library ported from `portfolio-site-v5-prime`'s `BlogImage` model, but URL-based (Cloudflare R2) instead of DB-stored bytes: upload once (with optional alt text), then reuse the same URL across multiple posts instead of re-uploading per post. `BlogImagesManager.vue` is the full CRUD page (upload/copy-URL/delete); `BlogImagePicker.vue` embeds a read-only picker directly in the post editor (a collapsible panel above the `MdEditor` in `BlogManager.vue`) — clicking a thumbnail inserts `![alt](url)` at the cursor via `md-editor-v3`'s exposed `insert()` method, clicking the copy icon just grabs the URL. There's no precedent for the in-editor picker in the old site (it only had the standalone management page with copy-URL, paste-manually workflow).

**Link previews are a real technical wrinkle, not just a mock-vs-real choice** — messaging apps (iMessage, Discord, WhatsApp, Slack) fetch a link's raw server HTML and read its `<meta>` tags; they do **not** execute JavaScript. `useDocumentMeta.ts` sets `document.title` and `og:`/`twitter:` meta tags reactively from the post's data, which is the correct thing to do for the browser tab and any client-rendered preview surface, but it **cannot** make sharing a `/blog/:slug` URL produce a real rich preview in those apps — that requires the *server* to return post-specific meta tags in the initial HTML response (e.g. Spring Boot serving a templated shell per slug, or detecting crawler UAs). `BlogController` now exists and serves real post data (`GET /api/blog/:slug`), but that's a plain JSON API, not server-rendered HTML with tags already baked in — the SSR piece itself is still a separate, not-yet-built concern. The post's own `title`/`excerpt`/`coverImage` (now real, API-backed data rather than sample JSON) are what a per-post server implementation would read, whenever it's built.

**Site settings** (`stores/siteSettings.ts`, backed by `public/data/site-settings.json`) — `siteTitle` (tab-title suffix, e.g. `"Blog - {siteTitle}"`), `defaultDescription`/`defaultShareImage` (meta/OG fallbacks for pages with no per-entity content: Home, Projects, Highlights, Contact), and `searchIndexingEnabled` (site-wide `robots` directive). `router/index.ts`'s `afterEach` applies these unconditionally on **every** navigation via a shared `applyBaselineMeta()` — deliberately not just once at boot, and deliberately not restored on unmount by the page that overrides them (`useDocumentMeta`, used by `BlogPostComponent`): since `afterEach` fires *before* the outgoing route's component unmounts, having the outgoing page clean up after itself would be redundant at best and a footgun at worst (whichever runs last wins); re-asserting the baseline unconditionally on the way *in* to every route is the only ordering that's actually reliable. `initSiteSettingsTracking()` (called from `main.ts`, after `app.use(pinia)`) re-applies the baseline for the current route once the settings JSON resolves, since that load is async and can complete after the initial `afterEach` already ran with fallback values. Editable via the admin's `/settings` page, still using a `localStorage`-override-over-baseline-JSON mock pattern (`public/data/site-settings.json`) — unlike Home/Highlights/Projects/Blog, which have since been wired to the real API, Site Settings has no backend model/controller yet and is the one outstanding mock in the app.

To validate the *mechanism* (not the production fix) without waiting for that backend work, `vite.config.ts` has a **dev-only** plugin (`blogPostMetaDevPlugin`, `apply: 'serve'`) that intercepts fresh (non-SPA-client-routed) requests to `/blog/:slug` and injects the same title/`og:`/`twitter:` tags server-side into the raw HTML, reading directly from `public/data/blog.json`. This makes the exact thing a crawler would see checkable locally with a plain `curl` (no JS involved), and — paired with a tunnel tool like `ngrok`/`cloudflared` pointed at the dev server — testable against real external tools (opengraph.xyz, Discord, iMessage) without deploying anywhere; `og:url`/`og:image` are built from the *actual request's* `Host` header, so they resolve correctly whether hit via `localhost` or a tunnel hostname. This plugin only ever runs under `vite dev` — a real static production build has no per-request server logic for it to hook into, so the equivalent still needs to exist server-side in Spring Boot eventually.

The same "server returns real tags/content, not just client-side JS" idea also generalizes toward **SEO**, with caveats: Google's crawler does execute JS (in a delayed second rendering pass), but Bing and most others don't reliably execute it at all, so server-rendered tags help everyone and are the *only* thing that works for non-Google crawlers and social unfurlers alike. What this doesn't cover is the actual article body — that's still client-rendered only, and genuine content SEO (versus just link-preview cosmetics) would need real SSR/prerendering of the post content, not just meta-tag injection. Natural next additions once the backend does this for real: `<link rel="canonical">`, JSON-LD `Article` structured data, and a generated `sitemap.xml`.

Gotchas hit building this:
- **`router.afterEach` runs *before* the destination route's component mounts**, per Vue Router's documented navigation resolution order (afterEach fires once navigation is confirmed; the `<RouterView>` DOM update — i.e. new component creation — happens after). `router/index.ts` uses `afterEach` to set a baseline `document.title` per route; `useDocumentMeta` used to also restore a captured "previous title" `onUnmounted`, but that fires *after* `afterEach` for whatever route you're navigating *to*, so it would silently stomp the correct new title back to a stale one (e.g. leaving a blog post straight for Home would leave the tab reading "Blog"). Fixed by not restoring the title on unmount at all — the next navigation's `afterEach` already guarantees the right one; `useDocumentMeta` only needs to clean up the `og:`/`twitter:` meta tags it injected, which no route guard handles.
- **`/blog/:slug` needs `BlogPostComponent` keyed on the slug** (`BlogPostView.vue` does `:key="route.params.slug"`), or navigating from one post directly to another reuses the same component instance (same matched route) instead of remounting — silently keeping the *previous* post's reactions/comments/`localStorage` keys and document meta alive instead of resetting for the new one.
- **`import hljs from 'highlight.js'` bundles all ~190 language grammars** — swapping to `highlight.js/lib/core` + explicit `registerLanguage()` calls per language actually used (`utils/markdown.ts`) dropped the production bundle from 1.3MB to ~455KB. Add a language import there before using it in a code fence, don't reach for the top-level package.
- **`@tailwindcss/typography`'s `.prose`/`.prose-invert` CSS vars are rebound onto our own tokens** (`main.css`, `--tw-prose-*`/`--tw-prose-invert-*`) rather than left at the plugin's default gray/blue palette, and both sets point at the *same* `--color-*` var()s rather than needing two physical palettes — `.dark` already swaps those, so `dark:prose-invert` fallout is free. `prose-pre`/`prose-code` are explicitly zeroed out (`prose-pre:bg-transparent prose-pre:p-0`, `prose-code:before:content-none`/`after:content-none`) since `renderMarkdown` (in `utils/markdown.ts`) already emits fully-styled `highlight.js` markup for code blocks — letting the plugin's own `pre`/`code` styling through too would double up backgrounds/padding and add unwanted inline-code backtick pseudo-elements.

`App.vue` uses a flex sticky-footer layout (`flex min-h-screen flex-col`) so `NavbarComponent` and `FooterComponent` always sit at the top/bottom of the viewport regardless of page content length. `<main>` is `flex w-full flex-1 justify-center` — because of this, a view's root element is a flex item and will **not** stretch to fill available width on its own; give it `w-full` explicitly (alongside any `max-w-*`) if it needs to actually reach that max width instead of shrink-wrapping its content (see `ContactComponent.vue`). Static assets (imported from the previous Next.js portfolio) live in `public/assets/`.

**Fonts** — carried over from the old portfolio, loaded via Google Fonts `@import` in `src/assets/main.css` and exposed as `@theme` tokens (which Tailwind 4 auto-generates matching `font-*` utilities for). Usage convention: `--font-salsa` (`font-salsa`) is the default for **most** UI text — applied to the whole `<nav>` in `NavbarComponent` and to `ContactComponent`'s form (labels, inputs, button); `--font-kalam` (`font-kalam`) is reserved for spots that want **more style** — the footer copyright and the Contact page subtitle. `--font-sans` (Geist) is only the raw fallback set on `body` in `base.css`, not meant to be reached for directly in components. `Charm` and `Pacifico` existed in the old site's CSS but were never actually used, so they were not ported. The font `@import` lines must stay **before** `@import 'tailwindcss'` in `main.css` — Tailwind's import expands into real CSS rules at that position, and a plain `@import` placed after any non-import rule in the compiled stylesheet is invalid per the CSS spec and gets silently dropped by the browser.

Also note: form controls (`input`/`textarea`/`button`) don't reliably inherit the ambient font in every browser even with Tailwind's preflight `font: inherit` rule — set the font utility explicitly on the control itself (as `ContactComponent.vue` does) rather than relying on inheritance from an ancestor.

**Color tokens** — `src/assets/main.css` defines a semantic palette via `@theme` (`--color-surface`/`-muted`, `--color-ink`/`-muted`, `--color-primary` (Sky), `--color-secondary` (Lime — reserved for CTAs only), `--color-warning` (Amber), `--color-danger` (Red)), each with light/dark values (dark overrides in the `.dark` block). All components (`NavbarComponent`, `FooterComponent`, `ContactComponent`) are built against these tokens now — use them (`bg-surface`, `text-ink`/`text-ink-muted`, `border-ink-muted/20`, etc.) instead of raw Tailwind `gray-*` classes for anything theme-aware. This matters beyond consistency: Tailwind v4's default `gray` scale is defined in OKLCH with a ~265° (blue-violet) hue, so it reads with a visible cool/blue tint next to the app's true-neutral `surface`/`ink` tokens — this caused a visible color mismatch in the Navbar/Footer dark mode before they were migrated.

**Theme default** — `useThemeStore` (`src/stores/theme.ts`) defaults to **light** on first visit (not the OS `prefers-color-scheme`); it only switches to dark if the user has explicitly toggled it before (persisted to `localStorage`).

**Cascade layers gotcha** — the global reset in `src/assets/base.css` (`box-sizing`/`margin` on `*`) must stay **inside** `@layer base`. Per the CSS Cascade Layers spec, unlayered styles form an implicit final layer that outranks every named layer regardless of selector specificity, so an unlayered `* { margin: 0 }` would silently override every Tailwind margin utility (`mb-*`, `mt-*`, etc.) app-wide. This exact bug happened once already — if margin/padding utilities mysteriously stop applying, check this first before assuming the utility class is wrong.

**Backend** — Spring Boot 4 with Spring MVC (`spring-boot-starter-webmvc`) and Spring Data JPA backed by PostgreSQL (configured on port 5430). Base package is `com.sresthaa`, split into three packages:
- `com.sresthaa.publicui` — all public-site content, each with its own model/repository/service/controller: `HomeContent` (+ nested `TimelineEvent`), `Project`, `Highlight` (renamed from `Gallery` mid-build — if old notes/branches reference `Gallery`, that's this), `Blog` + `BlogComment` + `BlogImage` (the reusable image library, admin-only despite living alongside public Blog endpoints), `ContactForm`. Every content type follows the same shape: a plain public `GET` for reads (`/api/home`, `/api/projects`, `/api/highlights`, `/api/blog`, `/api/blog/{slug}`, `/api/blog/{slug}/comments`) and `POST`/`PUT`/`DELETE /api/admin/...` for writes, gated by `SecurityConfig`'s blanket `/api/admin/**` rule (no per-endpoint auth annotations needed). `Blog` reactions (`ReactionVote` enum, `POST /api/blog/{slug}/reactions`) and `BlogImage` were added after the initial Blog CRUD, once comments/likes moved off their `localStorage` mocks.
- `com.sresthaa.admin` — everything auth-related: `AdminAccount` (single implicit admin — no multi-user/roles system), `WebAuthnCredential`/`TotpCredential`/`TotpBackupCode` (both 2FA paths — security keys and TOTP backup — are real, not mocked), `JwtService`/`JwtAuthenticationFilter`/`SecurityConfig` (short-lived JWT sessions), `AuthController`/`AccountController`/`WebAuthnController`/`TotpController`, plus `ImageUploadController` — the one shared upload endpoint (`/api/admin/uploads/image?category=...`) every content type's image fields go through, with `category` allow-listed per content type (`highlights`/`projects`/`blog`/`settings`).
- `com.sresthaa.storage` — `R2StorageService`, a thin wrapper around the AWS S3 SDK pointed at Cloudflare R2 (S3-compatible, `forcePathStyle`, region `"auto"` per R2's docs); `upload()`/`delete()` take/return full public URLs (not bucket keys) since that's what callers actually persist on their entities.

`HeartbeatController` is the only controller with no real dependencies (a leftover from the original empty scaffold). **Site settings/SEO is the one content type with no backend at all** — still a `public/data/site-settings.json` + `localStorage`-override mock in both frontends (see the Site settings paragraph above); real search-engine indexing (sitemap/canonical/SSR article body) is also explicitly deferred past initial launch, scoped down to link-preview meta tags only — see the Link previews paragraph above.

**Admin** — Astro 7, chosen for the admin panel over extending the Vue UI as a separate sub-project entirely. Vue components are mounted via `@astrojs/vue` (`client:load`/`client:only="vue"`); each `src/pages/*.astro` is a thin `Layout` + `NavbarComponent` + one page-level Vue component, same thin-wrapper-per-route shape as the UI's Views. Has its own copy of the UI's Tailwind config/color tokens/fonts rather than importing them — Astro `.astro` files and Vue SFCs are different formats, so there's nothing to directly share besides Vue components themselves (none currently are).

One manager component per content type, all following the same list/form-toggle CRUD shape (`view: 'list' | 'form'` local state, not separate routes — no client-side router in the admin): `HomeDashboard.vue` (`/home`, reorderable about-story/timeline lists via move-up/down, "reset to defaults"), `ProjectsManager.vue` (`/projects`), `HighlightsManager.vue` (`/highlights`), `BlogManager.vue` (`/blog`, includes the embedded `BlogImagePicker.vue`) + `BlogImagesManager.vue` (`/blog/images`, the full image-library CRUD page), `ContactMessagesComponent.vue` (`/contact`, read/mark-read/delete only, no message-content editing), `SiteSettingsComponent.vue` (`/settings` — the one remaining mock, see above). `AuthGate.vue` wraps every content page, redirecting to `/` (`LoginForm.vue`) without a valid session; `useApi.ts`'s `authFetch()` attaches the bearer token to every admin request and force-logs-out on a `401`.

`/account/*` is its own page group, real end-to-end: `AccountHub.vue` (index), `UsernameComponent.vue`, `PasswordResetComponent.vue`, `TwoFactorComponent.vue` (TOTP enrollment + QR provisioning), `SecurityKeysComponent.vue` (WebAuthn registration) — hitting `AccountController`/`TotpController`/`WebAuthnController` respectively.

Every image upload in the admin (cover images, highlight/project images, blog cover + inline + library images) goes through the same pipeline: `useImageProcessing.ts` (resize + re-encode to WebP via `<canvas>`) → `useImageUpload.ts`'s `uploadImage(blob, category)` → the shared `/api/admin/uploads/image` endpoint. `processImageUpload`/`processSingleImageUpload` return `Blob`s (for real upload), not data URLs — an earlier version of `processSingleImageUpload` returned a data URL for direct `localStorage` embedding, back before the upload endpoint existed; if you see a composable/component still doing that, it's stale and should be migrated to the `Blob` + `uploadImage` pattern.

**CSP gotcha** — `src/layouts/Layout.astro`'s Content-Security-Policy meta tag (prod-build only, see the comment there for why) needs `img-src` to include the R2 public host (`https://*.r2.dev`, or a custom domain if one replaces it) — this was missing for a while after R2 image storage was wired up, silently blocking every highlight/project/blog image preview in production builds only (the meta tag is dev-gated off, so it wasn't visible locally). If images mysteriously don't render in a prod build/preview but do in dev, check this first.

**Test environment** — UI tests run in jsdom. API tests are full `@SpringBootTest` + `MockMvc` integration tests per controller (`@AutoConfigureMockMvc`, `@Transactional` for rollback-per-test), not slices — one test class per public/admin controller (`HighlightControllerTest`, `ProjectControllerTest`, `BlogControllerTest`, `BlogImageControllerTest`, `ContactFormControllerTest`, `HomeContentControllerTest`, plus the admin-auth ones: `AccountControllerTest`, `AuthControllerTest`, `TotpControllerTest`, `WebAuthnControllerTest`, `ImageUploadControllerTest`). `R2StorageService` is `@MockitoBean`-mocked in every test that touches image fields (the real one talks to an actual Cloudflare bucket over the network) — follow that pattern for any new controller test involving uploads/deletes. Admin has no test setup yet.
